# 웹 투표 점유율 표시 설계

## 목표

앱의 투표 후보별 득표 표시 규칙을 웹의 모든 실제 투표 표시 화면에 동일하게 적용한다. 진행 중 투표는 일반 사용자에게 점유율을 표시하고, 관리자와 슈퍼관리자에게는 점유율 뒤 괄호 안에 원시 득표수를 함께 표시한다.

## 범위

- 투표 상세 후보 카드
- 투표 상세 TOP 3 포디움
- 투표 목록 카드의 미니 포디움
- 실제 화면에서 사용하는 공용 랭킹 및 투표 카드
- 위 화면의 실시간 또는 폴링 갱신 결과

후보별 점유율이 아닌 상세 헤더의 전체 투표 규모(`총 N표`)는 원시 득표수로 유지한다. 종료된 투표의 후보별 표시도 앱과 동일하게 기존 원시 득표수 형식을 유지한다.

## 표시 규칙

진행 중 투표의 후보별 점유율은 해당 투표의 삭제되지 않은 전체 후보 득표 합계를 분모로 계산한다. 검색 결과나 상위 3명만의 합계는 사용하지 않는다.

- 일반 사용자: `35.20%`
- 관리자 또는 슈퍼관리자: `35.20% (711,479)`
- 0표, null 득표수, 전체 합계 0: `—`
- `0 < 점유율 < 0.0001%`: `<0.0001%`
- 단일 후보가 전체 표를 가진 경우: `100.00%`
- 종료된 투표: 기존 형식인 `711,479` 또는 화면 문맥에 따른 `711,479 표`
- 예정 투표에서 노출되는 후보별 수치: `—`

점유율은 앱과 같은 유효숫자 규칙을 사용한다. 소수 자릿수는 최소 2자리, 최대 4자리이며 점유율 크기에 따라 다음처럼 정한다.

- 0.1% 이상: 소수점 2자리
- 0.01% 이상 0.1% 미만: 소수점 3자리
- 0.0001% 이상 0.01% 미만: 소수점 4자리

값은 버리지 않고 반올림한다.

## 구조

점유율과 득표수 문자열 생성은 UI와 인증 상태에 의존하지 않는 공통 순수 함수로 구현한다. 함수는 후보 득표수, 전체 득표수, 투표 상태, 관리자 여부를 입력받아 최종 표시 문자열을 반환한다. 이를 통해 모든 화면이 동일한 반올림, 빈 값, 관리자 괄호 규칙을 공유한다.

각 화면은 자신이 이미 가진 투표 상태와 전체 후보 데이터를 이용해 전체 합계를 계산한다. 관리자 여부는 기존 인증 훅의 `userProfile.is_admin` 또는 `userProfile.is_super_admin`에서 구한 뒤 명시적으로 하위 표시 컴포넌트에 전달한다. 공통 포맷터가 인증 스토어를 직접 읽지 않도록 해 서버/클라이언트 경계와 인증 로딩 상태를 숨기지 않는다.

실시간 또는 폴링으로 후보 득표수가 갱신되면 현재 전체 후보 배열에서 합계를 다시 계산하므로 후보 점유율과 관리자용 괄호 수치가 같은 렌더에서 함께 갱신된다.

## 컴포넌트 적용 원칙

- 공용 카드 컴포넌트에는 기존 호출부를 깨지 않는 선택적 표시 컨텍스트를 추가한다.
- 진행 중 목록은 전체 후보 합계와 관리자 여부를 미니 포디움 렌더러에 전달한다.
- 종료 목록은 기존 원시 득표수 UI를 유지한다.
- 상세 포디움과 후보 그리드는 상세 훅에서 계산한 동일한 전체 합계와 관리자 여부를 공유한다.
- 데모나 테스트 전용 화면은 실제 사용자 화면의 규칙을 검증하는 데 필요한 경우에만 갱신하며, 별도 기능을 추가하지 않는다.

## 오류 및 경계 처리

득표수가 null이거나 음수인 경우 표시 계산에서는 0으로 취급한다. 전체 합계가 0 이하이면 나눗셈을 하지 않고 `—`를 반환한다. 관리자 프로필이 아직 로드되지 않았거나 권한을 확인할 수 없으면 일반 사용자 표시를 사용해 원시 득표수가 불필요하게 노출되지 않도록 한다.

## 테스트 전략

TDD로 다음 순서대로 구현한다.

1. 공통 포맷터 테스트에 앱과 동일한 반올림 및 경계값을 추가하고 실패를 확인한다.
2. 최소 포맷터 구현으로 테스트를 통과시킨다.
3. 대표 목록 화면 테스트에서 진행 중 일반 사용자의 퍼센트와 관리자의 괄호 득표수를 검증한다.
4. 상세 후보 카드와 포디움 테스트에서 같은 규칙과 전체 후보 합계 사용을 검증한다.
5. 종료된 투표의 기존 득표수 표시가 유지되는 회귀 테스트를 실행한다.
6. 관련 Vitest 모음과 타입 검사를 실행한다.

## 제외 범위

- 앱에 있는 1위와 2위의 표 차이 말풍선
- 상세 헤더의 총 투표수 변경
- 투표 API 응답 구조 변경
- 데이터베이스 스키마 또는 집계 로직 변경
- 투표 카드의 레이아웃 및 시각 디자인 개편
